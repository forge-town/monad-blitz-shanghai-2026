Prove It
01 / 这项目到底是什么
项目理解版

这不是一个“AI 页面”，而是一个 AI 可信度机制。

Prove It 的核心不是“让 AI 回答问题”，而是“让多个 AI 同时回答同一个问题，然后把它们的结果拿去交叉验证，并用链上经济规则来奖惩”。

一句话理解

不是信任某一个 AI，而是让多个 AI 在同一任务里竞争、互相印证，并且把结果放到 Monad 上结算。

你现在要先记住的 4 个词
多 Agent
不是单模型输出
Commit-Reveal
先承诺再公开
Stake / Slash
有质押，有惩罚
Monad
并行执行更契合
你讲的时候最重要的角度

把它当成“AI 的 trust layer”来讲，而不是“一个用了 Claude 的 dApp”。

先别急着记页面，先把机制想明白。
脑内翻译

这个项目想解决的是：AI 会说，但谁来证明它说得对？Prove It 的答案是，让多个 AI 一起说，再把“谁更可信”变成链上可结算的规则。

问题
02 / 为什么值得做

它要解决的不是“AI 不够聪明”，而是“AI 不够可信”。

现在的问题

单点输出

一个 AI 给一个答案，这个答案本身没有内建的可信度机制。

人工复核

人工审核可以提升准确率，但不可能在大规模 agent 场景里持续跟上。

无责任

AI 说错了，通常没有直接代价。它没有“输的痛感”。

这个项目的回答

横向对照

让多个 agent 同时做同一题，不再依赖单个模型的自信程度。

先提交哈希

先 commit、后 reveal，减少“看别人答案再抄”的空间。

经济激励

一致的赚奖励，离群的被 slash，把“对错”变成有成本的博弈。

所以它的重点是信任机制，不是模型能力本身。
你自己要想通

如果没有 stake 和 slash，这个项目就只是“多个 AI 一起回答问题”。真正让它像协议而不是像普通应用的，是后面的经济规则。

机制
03 / 6 步流程

你可以把整个协议记成一个 6 步循环。

01 Register

注册 agent

agent 先上链注册，并带着 stake 进入系统。

02 Create

创建任务

任务有 reward pool、时间窗口、参与门槛。

03 Solve

多个 AI 同时求解

这里走的是 tRPC + Claude 的并行调用。

04 Commit

先提交承诺

不是直接交答案，而是先交结果哈希。

05 Reveal

再公开答案

之后 reveal 真正结果，合约核对是否匹配 commit。

06 Judge

判定共识并结算

一致者拿奖励，离群者被 slash，任务结束。

最关键的理解点

Prove It 的核心不是“谁答得最好”，而是“谁在一个可验证的流程里形成共识，谁就更值得信任”。

这页是整个项目的骨架。
你上台时的脑内顺序

stake 进场 → 任务发布 → 多 agent 同时答 → commit 防抄 → reveal 公开 → judge 决定共识 → reward/slash 结算。只要这条线清楚，你就不会讲乱。

为什么是 Monad
04 / 链的意义

这个项目放在 Monad 上，不只是“部署一下”，而是执行模型更贴合。

左边

多个 agent 需要在同一轮里交互

同一个任务里，多个 commit、多个 reveal，本来就是并发式的参与行为。

→
中间

Monad 的并行执行更契合这个场景

它不是那种“一个用户、一个交易、慢慢排队”的产品形状。

→
右边

所以 per-task 验证更现实

如果以后任务量大、参与 agent 多，这种模式在 Monad 上更自然。

正确讲法

不要只讲 TPS。要讲“这个协议本来就长得像多方并发参与，所以 Monad 更像它应该生长的地方”。

错误讲法

不要变成纯基础设施背诵。评委更关心的是：为什么你的产品机制和 Monad 是匹配的。

讲 Monad 时，要从产品结构出发，不要只报参数。
一句能记住的话

这是个“多 agent 在同一轮里一起动”的协议，所以 Monad 的并行执行不是锦上添花，而是更贴形状。

代码现实
05 / 现在到底做成了什么

真实已经存在的部分

合约

已经有：`packages/contracts/src/AgentTrust.sol`，里面有 agent 注册、任务创建、commit、reveal、judge、reward、slash 等核心状态机。

AI 路由

已经有：`solveParallel` 和 `judge`，会并行调用 3 个 Claude 配置来做题和裁决。

前端骨架

已经有：`main`、`arena`、`demo`、`leaderboard`、`stats` 等页面，钱包连接和合约 hooks 也接上了。

交互思路

已经有：从“创建任务”到“AI 输出”到“链上结算”的完整叙事路径。

还没完全做完的部分

有些页面还是占位页。 例如 `agents.tsx` 和 `results.tsx` 目前基本还是 placeholder。

有些展示数据是 mock

比如 leaderboard、stats 页面里有一部分是为了产品表达而先写的假数据，不是全部实时链上数据。

你应该怎么理解完成度

这是一个 协议机制已经成形、产品包装还没完全封口 的项目，而不是一个所有页面都 fully production ready 的项目。

这页会帮你避免自己把项目讲过头。
最重要的判断

现在最硬核的部分是“机制和链上流程”，不是“所有视图都打磨完了”。所以 pitch 时重心要放对。

架构
06 / 你要怎么向自己解释这套系统

它本质上是三层：前端、AI 协调层、链上结算层。

前端

TanStack Start + wagmi

前端负责让用户看到任务流程、连钱包、点按钮、观察 log、看 leaderboard / stats 这些产品界面。

AI 协调层

tRPC + Claude

这一层负责让多个 agent 同时做题，并把结果交给 judge 形成一个共识判断。

链上层

AgentTrust 合约

这一层不是“生成答案”，而是负责规则、公平、状态转换、奖惩和最终可信结算。

用一句技术话概括
前端发起任务 → tRPC 并行调用多个 agent → 前端通过 wagmi 调合约提交 commit 进入 reveal → judge 形成共识簇 → submitJudgment 在 Monad 上完成 reward / slash
你可以把它理解成“AI 负责产出，合约负责信任”。
一句很好记

模型负责给答案，协议负责决定“这个答案值不值得被相信”。

页面理解
07 / 每个页面在项目里扮演什么角色

你现在打开的这些页面，不是平级的，它们承担的作用不同。

页面 它在讲什么 完成度理解
`main.tsx` 世界观入口。告诉用户这是一个 AI capability proof / trust layer 项目。 偏品牌与入口。
`arena.tsx` 最核心的产品页面。这里最像“真实协议回合”的操作界面。 最值得拿来讲主流程。
`demo.tsx` 把复杂机制拆成一步一步的可演示流程，更适合给人看懂。 适合 pitch 演示。
`leaderboard.tsx` 表达这个协议未来会形成 agent reputation / ranking 层。 有产品价值，但有 mock 数据。
`stats.tsx` 表达协议经济飞轮、TVL、奖励和 slash 的视图。 偏概念展示，数据未必都实时。
`agents.tsx` / `results.tsx` 这些还没有完整承载叙事。 当前别把它们当主角讲。
最实用的结论

如果你只能带评委看一个页面，就看 `arena` 或 `demo`，因为那里最接近协议本体。

页面不是越多越好，关键是你知道哪页最能代表项目。
你自己看项目时

把 `arena` / `demo` 当“核心机制”，把 `leaderboard` / `stats` 当“未来产品层”，把 placeholder 页当“还没补齐的边缘功能”。

风险点
08 / 你必须心里有数的地方

这个项目现在有几个必须知道的“口径风险”。

风险点 现状 你应该怎么理解
命名不统一 仓库里同时有 `Prove It` 和 `AgentTrust` 两套叫法。 最合理的理解是：`Prove It` 是产品名，`AgentTrust` 更像合约 / 协议实现名。
地址不完全统一 不同文件里出现过不同合约地址。 说明项目还在快速迭代，配置还没完全收口。
judge 仍偏 MVP judge 现在依然是 AI-backed judge，不是完全去中心化裁判市场。 这意味着现在的重点是“验证主机制可行”，不是所有角色都已经去中心化。
部分视图是假数据辅助表达 leaderboard / stats 混合了 live count 和 mock 展示数据。 这说明产品叙事在前，数据产品化仍在补。
别把这些当坏事

这些不是致命缺点，它们更像黑客松项目的典型状态：核心机制先跑通，命名、配置、展示层还在收敛。

你越清楚这些，越不会被问住。
自己心里要稳

项目真正有价值的是“信任机制的雏形已经写出来了”，不是“每个角落都已经 polished”。

价值判断
09 / 这个项目最强的地方到底在哪

它最强的不是界面，而是把 AI 可信度问题“协议化”了。

强点 1：问题抓得准

现在 AI agent 很多，但“谁来验证 agent 质量”这个问题非常真实，而且比“再做一个 agent 应用”更有基础设施意味。

强点 2：机制是闭环的

多 agent、commit-reveal、judge、reward/slash，这几个部分不是零散点子，而是能连起来形成一套闭环。

强点 3：和 Monad 有自然结合

不是“任何链都一样”，而是这个产品天然适合多方并发参与，这让 Monad 的叙事更成立。

中期潜力

它未来可以延伸到 code review、translation、fact-checking、analysis 等任何“多个答案可以相互比较”的任务。

一句最本质的判断

Prove It 不是在做“更强的 AI”，而是在做“更可信的 AI 结果市场”。

这页是你建立信心的一页。
如果你只记一个核心卖点

“它把 AI 输出质量从一个黑盒体验，变成了一个可以对照、可以博弈、可以链上结算的过程。”

总结
10 / 你现在应该怎么理解这个项目

Prove It 是一个 AI 输出可信度协议 MVP。

你可以把它理解成：用多 agent 交叉验证 + commit-reveal + stake/slash，把“AI 说得对不对”这件事变成一套可以在 Monad 上运行的规则。

最终理解模板

它已经证明了协议主干是可写、可演示、可叙事的；剩下要继续补的是产品收口、数据一致性、页面完成度和更成熟的 judge / reputation 设计。

你现在最该记住的 4 句话

产品名叫 Prove It。

2

核心不是 UI，而是多 agent + 经济惩罚的 trust mechanism。

3

最该讲的是 `arena / demo` 里的那一轮完整协议流程。

现在最真实、最硬的部分是机制和链上状态机，不是所有页面都做完了。
